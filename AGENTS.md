# AGENTS.md

## 1. Назначение проекта

Проект является дипломным учебным проектом по специальности программирования.

Цель проекта — разработать веб-сервис для анализа статистики IT-вакансий. Сервис должен помогать пользователю понять:

- какие навыки наиболее востребованы для выбранной IT-специальности;
- какие навыки наиболее востребованы для выбранного грейда;
- насколько конкурентным является выбранное направление;
- в каком направлении пользователю стоит развиваться.

Проект должен быть реализован как демонстрируемый MVP, а не как полноценный промышленный сервис.

Основной приоритет — получить рабочую, понятную и защищаемую версию проекта в ограниченные сроки.

---

## 2. Ограничения проекта

На проект выделено ограниченное количество времени: примерно 6 часов в неделю в течение 4 недель.

Поэтому при разработке нужно избегать избыточной сложности.

Не нужно реализовывать в первой версии:

- регистрацию и авторизацию пользователей;
- личный кабинет;
- админ-панель;
- сложную систему ролей;
- React, Vue, Angular или другой сложный frontend-фреймворк;
- сложный ML/NLP;
- прогнозирование рынка труда;
- полноценную рекомендательную систему;
- сложную CI/CD-инфраструктуру;
- микросервисную архитектуру с очередями сообщений;
- Kubernetes;
- автоматическое масштабирование;
- сложную observability-инфраструктуру.

Если появляется идея дополнительного функционала, её нужно записывать в документацию или TODO, но не реализовывать без явной необходимости.

Главный критерий выбора решения: оно должно помогать быстрее получить рабочий дипломный MVP.

## 3. Основной пользовательский сценарий

Пользователь открывает сайт и выбирает:

- специальность;
- грейд.

После этого пользователь видит:

- количество найденных вакансий;
- коэффициент конкурентности;
- уровень конкурентности в текстовом виде;
- топ востребованных навыков;
- график востребованности навыков;
- дату последнего обновления данных;
- источник данных.

Пример сценария:

1. Пользователь выбирает специальность `Backend-разработчик`.
2. Пользователь выбирает грейд `Junior`.
3. Сервис показывает, что найдено 42 вакансии.
4. Сервис показывает коэффициент конкурентности, например `0.77`.
5. Сервис показывает топ навыков: `SQL`, `Git`, `Docker`, `C#`, `ASP.NET Core`.
6. Пользователь понимает, какие навыки стоит изучать для выбранного направления.

## 5. Общая архитектура

Система состоит из четырёх основных частей:

```text
[Browser]
    ↓ HTTP
[Frontend]
    ↓ HTTP
[ASP.NET Core Backend API]
    ↓ SQL
[PostgreSQL]
    ↑ SQL
[Python Parser]
```

Frontend не должен обращаться к базе данных напрямую.

Frontend получает данные только через ASP.NET Core Backend API.

Python Parser может напрямую записывать данные в PostgreSQL, так как это учебный MVP с ограниченным сроком разработки.

В документации нужно отметить, что более правильный production-вариант выглядел бы так:

```text
[Python Parser] → [Backend API] → [PostgreSQL]
```

Однако для текущего MVP допустима схема:

```text
[Python Parser] → [PostgreSQL] ← [Backend API]
```

## 7. Подход к разработке

Проект разрабатывается сверху вниз.

Порядок мышления:

```text
Пользовательский экран
↓
Необходимые данные на экране
↓
Контракт API
↓
Структура backend
↓
Структура БД
↓
Parser для наполнения БД
```

Сначала нужно понять, что должен видеть пользователь, и только потом проектировать backend, БД и parser.

При разработке нового функционала сначала нужно ответить на вопросы:

1. Где это будет отображаться на frontend?
2. Какие данные нужны frontend?
3. Какой endpoint должен отдать эти данные?
4. В каких таблицах эти данные должны храниться?
5. Как parser должен получить или рассчитать эти данные?

## 10. Коэффициент конкурентности

В рамках MVP коэффициент конкурентности рассчитывается косвенно.

Базовая формула:

```text
competitiveness = 1 - vacanciesForGrade / totalVacanciesForSpeciality
```

Где:

- `vacanciesForGrade` — количество вакансий по выбранной специальности и грейду;
- `totalVacanciesForSpeciality` — общее количество вакансий по выбранной специальности.

Пример:

```text
Всего Backend-вакансий: 180
Junior Backend-вакансий: 42

competitiveness = 1 - 42 / 180 = 0.77
```

Интерпретация:

```text
0.00–0.30 — низкая конкуренция
0.31–0.60 — средняя конкуренция
0.61–1.00 — высокая конкуренция
```

Важно: в документации нужно честно указать, что это косвенный показатель, так как сервис не имеет данных о количестве соискателей.

## 13. Parser

Parser должен быть отдельным Python-сервисом.

Основные задачи parser-а:

- получить вакансии из источника данных;
- извлечь название вакансии;
- извлечь описание вакансии;
- определить специальность;
- определить грейд;
- найти навыки в тексте вакансии;
- сохранить вакансии в PostgreSQL;
- сохранить связи между вакансиями и навыками;
- писать понятные логи.

Parser должен иметь запасной режим работы с локальным JSON-файлом.

Это нужно на случай, если источник данных временно недоступен или парсинг сломался.

Пример:

```text
parser/data/raw_vacancies.json
```

В демонстрационном режиме можно использовать заранее сохранённый набор вакансий.

## 14. Правила для Python-кода

В Python-коде обязательно:

- использовать логирование;
- писать docstring для функций, классов, методов, атрибутов классов и глобальных переменных;
- docstring писать на русском языке;
- комментарии писать на русском языке;
- сложные и неочевидные места пояснять комментариями;
- проектировать код так, чтобы его было легко тестировать;
- выносить работу с внешними источниками в отдельные классы/функции;
- не смешивать парсинг, обработку данных и сохранение в БД в одной большой функции.

Рекомендуемая структура parser-а:

```text
parser/
├── vacancy_parser/
│   ├── __init__.py
│   ├── config.py
│   ├── logger.py
│   ├── models.py
│   ├── source_client.py
│   ├── skill_extractor.py
│   ├── grade_detector.py
│   ├── repository.py
│   └── service.py
├── main.py
└── requirements.txt
```

Пример разделения ответственности:

- `source_client.py` получает данные из внешнего источника;
- `skill_extractor.py` ищет навыки в тексте;
- `grade_detector.py` определяет грейд;
- `repository.py` работает с PostgreSQL;
- `service.py` связывает шаги parser-а в один сценарий.

## 15. Правила для C# / ASP.NET-кода

Backend должен быть понятным и не перегруженным.

Рекомендуется использовать слои:

- Api;
- Application;
- Domain;
- Infrastructure.

Но если из-за сроков это станет слишком тяжело, допустимо упростить структуру.

Главное — не писать всю бизнес-логику в контроллерах.

Контроллеры должны:

- принимать HTTP-запрос;
- вызывать сервис;
- возвращать DTO.

Сервисы должны:

- рассчитывать статистику;
- формировать DTO для frontend;
- скрывать детали работы с БД.

Repository или DbContext должны:

- работать с PostgreSQL;
- получать данные;
- сохранять данные при необходимости.

Для публичных классов и методов желательно писать XML-комментарии на русском языке.

Логирование должно использовать стандартный `ILogger<T>`.

## 17. Docker Compose

Проект должен запускаться через Docker Compose.

Ожидаемые сервисы:

- `frontend`;
- `backend`;
- `parser`;
- `postgres`.

Примерная схема:

```yaml
services:
  postgres:
    image: postgres:16

  backend:
    build: ./backend
    depends_on:
      - postgres

  parser:
    build: ./parser
    depends_on:
      - postgres

  frontend:
    build: ./frontend
    depends_on:
      - backend
```

Конкретный `docker-compose.yml` нужно писать отдельно.

Секреты и пароли не должны попадать в репозиторий.

Нужно использовать `.env` и `.env.example`.

## 20. Что считать готовым MVP

MVP считается готовым, если:

- проект запускается через Docker Compose;
- frontend открывается в браузере;
- пользователь может выбрать специальность и грейд;
- frontend получает данные из backend;
- backend получает данные из PostgreSQL;
- parser может наполнить PostgreSQL данными или загрузить данные из JSON;
- отображается топ навыков;
- отображается коэффициент конкурентности;

При выполнении задач агент должен:

1. Сначала понять, к какому компоненту относится задача: frontend, backend, parser, database, docs или deployment.
2. Не менять несколько независимых частей проекта без необходимости.
3. Делать изменения небольшими и проверяемыми.
4. Не добавлять сложные технологии без явной причины.
5. Не переписывать архитектуру без необходимости.
6. Сохранять совместимость с Docker Compose.
7. При изменении API обновлять документацию.
8. При изменении структуры БД учитывать, что её используют backend и parser.
9. При изменении формата ответа backend учитывать frontend.
10. Добавлять логи в backend и parser.
11. Писать код так, чтобы его можно было тестировать.
12. Не хранить секреты, токены и пароли в коде.
13. Не использовать реальные персональные данные пользователей.
14. Не перегружать MVP второстепенным функционалом.

## 23. Важные архитектурные решения

### Решение 1. Один репозиторий

Проект хранится в одном репозитории, потому что это ускоряет разработку и упрощает запуск.

### Решение 2. PostgreSQL в Docker

База данных запускается в Docker-контейнере через Docker Compose.

### Решение 3. Backend владеет схемой БД

Миграции и основная структура БД относятся к backend.

### Решение 4. Parser пишет в PostgreSQL напрямую

Для MVP parser может напрямую записывать данные в БД.

### Решение 5. Frontend работает только через Backend API

Frontend не знает о структуре БД и не подключается к PostgreSQL.

### Решение 6. Frontend сначала проектируется на mock-данных

Это позволяет двигаться сверху вниз: от пользовательского интерфейса к API, БД и parser-у.

## 24. Термины проекта

### Специальность

Направление IT-вакансии, например Backend-разработчик или DevOps-инженер.

### Грейд

Уровень квалификации, например Junior, Middle или Senior.

### Навык

Технология, инструмент или компетенция, найденная в вакансии.

### Вакансия

Единица данных, полученная из внешнего источника.

### Коэффициент конкурентности

Косвенный показатель, который показывает предполагаемую сложность входа в выбранное направление.

### Dashboard

Главный экран сервиса, на котором пользователь видит статистику по выбранной специальности и грейду.

## 25. Итоговая цель

Итоговый проект должен выглядеть как небольшой, но законченный веб-сервис.

Главная ценность проекта — не в количестве функций, а в том, что пользователь может открыть сайт, выбрать направление и получить понятную аналитику по востребованным навыкам.
