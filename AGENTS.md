## 1. Общая информация о проекте

Проект: **Веб-сервис анализа рынка IT-вакансий**

Цель:
Дать соискателю понятную картину рынка:

* какие навыки востребованы;
* насколько высокая конкуренция;
* какие требования предъявляют работодатели;
* как выбрать направление развития.

Ограничения:

```text
~4 недели
~24 часа разработки
1 разработчик
```

Главный приоритет:

```text
Сделать простой, рабочий MVP с полезной аналитикой.
```

---

## 2. Планируемая архитектура

```text
[Browser]
    ↓
[Flask (backend + HTML)]
    ↓
[PostgreSQL]
    ↑
[Parser (Python)]
```

Компоненты:

* Flask — backend + страницы (Jinja2)
* Parser — сбор данных (вакансии + агрегаты резюме)
* PostgreSQL — хранение данных

## 5. Принципы разработки

### 1. Простота

```text
Минимум сложности.
Лучше простое решение, чем “правильное, но долгое”.
```

---

### 2. MVP > идеальная архитектура

```text
Не тратить время на:
- микросервисы
- сложные паттерны
- избыточную абстракцию
```

---

### 3. Минимальный frontend

```text
Jinja2 + HTML + Chart.js
Без React/Vue
```

---

### 4. Парсинг — зона риска

```text
Если парсинг нестабилен:
- использовать тестовые данные
- не блокировать разработку
```

---

### 5. Безопасность данных

```text
НЕ сохранять персональные данные резюме
Только агрегированные значения
```

---

### 6. Делать fallback

```text
Нет данных → не показывать метрику, а не ломать страницу
```

---

## 6. Definition of Done (MVP)

Проект готов, если:

```text
- есть веб-страница с фильтрами
- можно выбрать специальность
- показывается:
    - количество вакансий
    - количество резюме
    - коэффициент конкурентности
- есть минимум 2–3 графика
- есть таблица топ навыков
- данные берутся из БД
- всё запускается локально через docker-compose
```

---

## Главный принцип

```text
Сервис должен помогать принять решение:

"куда идти и что учить"
```

А не просто показывать красивые графики.

## 7. Текущий статус реализации

Реализован первый UI-first срез MVP:

```text
Flask + Jinja2 + Chart.js
данные пока из реалистичного mock-провайдера
PostgreSQL и парсер еще не подключены
```

Что уже есть:

```text
- маршрут GET /
- русскоязычный dashboard
- фильтр по специальности
- карточки:
    - количество вакансий
    - количество активных резюме
    - коэффициент конкурентности
- графики:
    - распределение вакансий по грейдам
    - распределение требуемого опыта
    - распределение формата работы
- таблица топ навыков
- fallback для специальности без вакансий
- базовые pytest-тесты
```

Ключевой технический принцип текущей реализации:

```text
UI зависит от контракта app.analytics.get_dashboard_data,
а не от конкретного источника данных.
```

Это нужно сохранить при подключении БД: заменить источник данных, но не переписывать шаблон без необходимости.

## 8. План реализации UI-first MVP

### Этап 1. UI-first срез

Статус: выполнено.

Реализовать тонкое Flask-приложение:

```text
app/              # Flask app + analytics contract
templates/        # Jinja2 dashboard
static/css/       # стили dashboard
static/js/        # Chart.js и инициализация графиков
tests/            # тесты маршрута и analytics contract
requirements.txt
README.md
```

Требования к первому срезу:

```text
- UI на русском языке
- чистый аналитический стиль
- реалистичные mock-данные
- рабочий фильтр по специальности
- остальные фильтры видны как следующий этап
- no data fallback вместо поломки страницы
```

### Этап 2. Подключение PostgreSQL

Следующий приоритетный этап.

Нужно добавить:

```text
- docker-compose.yml с Flask и PostgreSQL
- конфигурацию подключения через environment variables
- минимальную схему БД
- seed-данные для локального запуска
- DB-backed реализацию analytics contract
```

Минимальные сущности:

```text
vacancies:
  - specialty
  - grade
  - skills
  - experience
  - work_format
  - salary fields optional

resume aggregates:
  - specialty
  - grade optional
  - active_resume_count
  - collected_at
```

Важно:

```text
НЕ сохранять персональные данные резюме.
Хранить только агрегаты.
```

### Этап 3. Парсер и импорт данных

Добавить parser/import слой после стабилизации схемы:

```text
- parser пишет вакансии и агрегаты резюме в PostgreSQL
- при нестабильном парсинге использовать seed/mock данные
- падение парсера не должно ломать dashboard
```

### Этап 4. Расширение аналитики

Добавлять только после рабочего DB-backed MVP:

```text
- фильтр по грейду
- фильтр по навыку
- фильтр по формату работы
- зарплатная аналитика только если данных достаточно
```
